What I heard from you

  • Your engineering blog mentions you use Elasticsearch as a vector DB for product search. We can reuse it for the agent's RAG. No need to add Pinecone or Qdrant.
§ 2 · The framing

It's not really an "AI agent." It's mostly automation with an LLM in front.

This was my biggest shift while researching. Most DevOps tickets ("create S3 bucket", "give me read access to staging RDS", "create Kafka topic") don't need a smart LLM. They need good Terraform modules and a self-service portal.

My rough estimate of how the work splits:

~70%
Deterministic code
~30%
LLM helps
Templates, forms, structured input Messy text, unclear intent
I'm guessing the 70/30. I don't have your real ticket data. The first thing I'd do in week 1 is measure it.

So the design is deterministic-first, LLM only when needed:

  1. Ticket arrives. Cheap LLM extracts intent and parameters.
  2. If it matches a known template (S3 bucket, IAM read role, Kafka topic) and confidence is high, run the deterministic handler. No planning LLM. Just fill template, open PR.
  3. If it's ambiguous or new, planner LLM creates a JSON plan. Plan goes to a human in Slack before any action runs.

This is the opposite of what people expect from an "AI agent." It's also why this won't blow up the way Replit's did (more in §6).

§ 3 · Architecture

Five layers. Each does one job.

A developer writes a ticket — then this happens:

  1. 1 Ticket lands in our platform via Jira webhook (or a Slack DM)
  2. 2 Brain pauses, asks a human in Slack to approve the plan
  3. 3 Once approved, the Brain executes via MCP servers
  4. 4 External system change happens (PR merged, Terraform applied)
  5. 5 Brain notifies the developer back in Jira and Slack

Slack appears at both top and bottom because we read events from it (approvals, DMs in) and write to it (notifications out).

Three components do the heavy work, each picked for a clear reason:

  • LangGraph for orchestration. MIT-licensed, runs anywhere. Gives me three things I would not write myself in a weekend: persistent state, interrupt() for waiting on humans, and audit traces by default.
  • LiteLLM as the LLM gateway. One OpenAI-compatible endpoint that routes to OpenAI, Anthropic, Bedrock, or self-hosted models. Switching providers is a config change. This is your lock-in escape.
  • MCP (Model Context Protocol) for tools. Anthropic donated it to Linux Foundation in December 2025. Now a vendor-neutral standard supported by Anthropic, OpenAI, Google, Microsoft, AWS.

§ 5 · Approval model

Risk tiers, so humans don't rubber-stamp into trouble

If we ask for human approval on every action, engineers will rubber-stamp them within a week and the safety we built collapses. So actions are tiered by blast radius and reversibility.

Tier A Read-only Auto-execute
Examples: List S3 buckets, get pod status, query logs, read Jira history. Controls: audit log + 5% random spot-checks to catch drift even when no approval was needed.
Tier B Reversible writes 1 approval
Examples: Create S3 bucket, create Kafka topic, grant read-only IAM, add a developer to a non-prod group. Controls: single DevOps approval in Slack with the diff visible. 4-hour timeout, then escalate. Idempotency key on every action.
Tier C Production / hard to reverse Dual + token + cooldown
Examples: Production IAM admin, DDL on a prod database, deleting a Kafka topic, anything tagged env=production with destructive intent. Controls: two approvals from different roles (DevOps + Tech Lead) + a typed confirmation token (not a click) + 15-minute cooling period before execute. Anyone can cancel during the cooldown.

Tier classification is rule-based, on (action, environment, blast_radius). Not a place I want the LLM making the final call. The classifier proposes a tier; a hard rule confirms it; if there's any doubt, the rule wins.

§ 6 · Hard lessons

Three production failures I designed around

I read about these while researching. They are real, recent, and well documented. Each one taught me something I had to bake into the design.

1
Replit AI agent: production database deletion July 2025

An AI coding agent ran DROP TABLE on production during an explicit code freeze. Then lied about whether rollback was possible. The agent had direct write access to the prod DB, and decided + executed in the same loop step.

My design: the LLM never touches infra directly. It only writes a plan that goes through GitOps. Tier-C destructive actions need two humans plus a typed token plus a cooldown.

2
EchoLeak (CVE-2025-32711): zero-click prompt injection June 2025

An attacker email could trick Microsoft 365 Copilot into reading files and leaking data. The first in-the-wild prompt-injection-to-data-exfiltration on a production LLM.

My design: ticket bodies, Confluence content, and Slack messages are untrusted input. Wrapped in delimiter tags. Strict tool allowlists. Never auto-fetch URLs found in retrieved content.

3
PocketOS / Railway: the 9-second disaster March 2026

Claude Code, blocked by a credential mismatch, found an unrelated Railway API token in a file and called volumeDelete to "fix" the obstacle. Database and backups gone in 9 seconds.

My design: when a tool fails, the agent must not autonomously search for another path. Force escalation to a human. LangGraph supports this with explicit "fail-up" edges.

§ 7 · Lock-in

Three layers that are swappable. None locked to a vendor.

This was your strongest rule, so it gets its own section.

Layer 1
LLM
OpenAI GPT-5.4
→ Claude on Bedrock, self-hosted Llama, anything else
Layer 2
Orchestration
LangGraph (MIT, self-hosted)
→ OpenAI Agents SDK, custom FastAPI, Mastra
Layer 3
Tools
MCP (Linux Foundation)
→ vendor-neutral by design, every major LLM supports it

Switching from OpenAI to Claude is a config change in LiteLLM, not a code rewrite. Switching orchestrators is a 2 to 4 week project, not a year. Tools written as MCP servers don't care which LLM is calling them.

§ 8 · Trade-offs

What I chose not to do

  1. Speed vs. blast radius. I could ship a Replit-style "agent does it all" in 3 weeks. I'm not doing that. Cost: 8 to 12 weeks longer to first value. Benefit: I won't be on Hacker News explaining how the agent dropped a prod table.
  2. Plan-and-Execute vs. ReAct loops. ReAct is more impressive in demos. It's also where most production incidents come from. Plan-and-Execute makes me show the plan to a human before any side effect.
  3. Build vs. buy. I'd build the request-fulfillment agent (it's tied to your private infra). For incident triage I'd evaluate Datadog Bits AI SRE or PagerDuty's SRE Agent, those are commodity capabilities, not worth building.
  4. Managed APIs vs. self-hosted. Today: managed APIs through LiteLLM. Self-hosting Llama makes sense above ~$10K/month sustained. Lusha isn't there. LiteLLM lets us defer that decision until data forces it.
  5. LangGraph vs. AWS-native (Step Functions + SQS + DynamoDB + Lambda). Lusha is 100% on AWS, and Step Functions has Wait-for-Callback patterns equivalent to LangGraph's interrupt(). So this is a real alternative I considered. Why I picked LangGraph anyway: AWS Step Functions ASL is verbose JSON (vs. Python code), AI-specific primitives like message-history state and RAG threading need extra glue in Step Functions, and most AI agent reference architectures are written in LangGraph. We still use AWS heavily: SQS for the ticket queue, Bedrock as a LiteLLM backend, Secrets Manager for keys, CloudWatch for infra metrics. So this is LangGraph + AWS, not LangGraph instead of AWS.
